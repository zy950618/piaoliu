# 产品规则

## 核心玩法

首版包含漂流瓶、游戏、树洞、广场、附近的人、消息中心、会员中心、钱包、认证收益、用户中心和后台看板。

底部主导航固定为五项：

- 瓶子
- 广场
- 游戏
- 树洞
- 我的

首页可以作为冷启动推荐页存在，但不放在底部 Tab。我的页承载个人中心能力。

## 今日次数

| 类型 | 基础次数 | VIP 加成 | 扣除时机 |
| --- | ---: | ---: | --- |
| 捞瓶子 | 5 | 5 | 成功捞到真实瓶子后 |
| 扔瓶子 | 3 | 3 | 发布成功并进入审核队列后 |
| 真心话 | 3 | 3 | 成功抽到题目后 |
| 大冒险 | 3 | 3 | 成功抽到任务后 |
| 树洞发布 | 2 | 2 | 发布成功并进入审核队列后 |

点击入口、进入页面、打开弹窗均不扣次数。

漂流瓶页的“捞”和“扔”入口必须一直可见，并用单独角标显示对应剩余次数。

## 广告、签到、拉新

- 完整观看 1 次激励视频后，五类今日次数各 `+1`。
- 广告冷却默认 30 分钟，前端显示倒计时。
- 签到一周循环奖励：`10, 10, 30, 10, 10, 30, 100` 漂流币。
- 签到只放在“我的中心”，不放在主页。
- 拉新赠会员：用户使用邀请码完成注册并通过基础风控后，邀请人累计达到配置人数可领取会员天数。
- 拉新奖励必须防刷：设备、手机号、支付账号、IP、行为轨迹都进入风控。

## 关系链

- 捞到瓶子后可以回应、关注、申请好友、送礼、举报/拉黑。
- 捞到瓶子后的详情必须展示作者头像、昵称、VIP、性别、年龄段、城市和认证标签。
- 禁止无上下文冷启动骚扰陌生人。
- 允许明确互动上下文内的陌生人私聊；好友关系不是私聊唯一门槛。
- 漂流瓶场景中，B 捞到 A 的瓶子并有效回应后，A 回复该回应或点击“继续聊/私聊”，双方可开启带来源的上下文私聊，无需先加好友。
- 广场/树洞场景中，B 评论 A 的内容后，A 回复该评论或点击“继续聊/私聊”，双方可开启带来源的上下文私聊，无需先加好友。
- 游戏/房间/扩列场景中，双方进入同一房间或匹配上下文，并完成明确互动或确认后，可开启临时私聊。
- 上下文私聊必须保留 `source_type`、`source_id`、双向回应/确认证据、频控、拉黑、举报、风控和审计记录。
- 好友关系用于长期关系沉淀、更多资料可见性、更低频控限制和长期消息优先级。
- 关注用于后续关注流、广场动态和回访，不等于好友。
- 黑名单用户不能再给我发好友申请、评论、礼物或查看我的私密内容。

## 女性友好与认证

- 认证流程包含人脸活体、男女识别、人工复核。
- 性别识别结果不能只靠模型自动通过，必须保留人工复核状态。
- 女性认证通过后才可开启私密照片收益、礼物收益和魅力值提现。
- 附近的人默认只展示粗略距离，不展示精确定位。
- 无明确互动上下文的陌生人互动默认限制为关注、好友申请、送礼和举报。

## 金币、收益和魅力值提现

- 充值金币：用户充值获得，只能用于查看照片、送礼物、购买权益，不可提现。
- 收益金币：来自私密照片查看、礼物收入、活动分成。
- 魅力值：收益金币按平台配置转换为魅力值，提现以魅力值为门槛和换算单位。
- 默认提现门槛：`1000` 魅力值。
- 默认换算：`100` 魅力值 = `1` 元。
- 提现申请进入人工审核和风控审核。
- 异常收益、被举报内容、诱导充值、疑似欺诈会冻结魅力值和收益。
- 私密照片采用 AI 智能审核优先、风险分级、人工复核兜底。
- 低风险、高置信、非露骨、无敏感隐私、无未成年人疑似、无诈骗导流内容可自动通过。
- 中风险、低置信、边界内容进入人工复核。
- 高风险内容自动拒绝、冻结或进入更严格复核。
- 收益只允许来自审核通过且未冻结的内容；审核中、拒绝、冻结、申诉待处理内容不得产生收益。
- 所有审核必须保留模型标签、置信度、风险等级、自动动作、人工复核记录、收益冻结/解冻记录和审计日志。

## 页面分工

- 漂流瓶页：扔瓶、捞瓶、瓶子详情、回应、关注、好友申请、送礼、举报。
- 扔瓶时可按本次发布选择“默认/男/女”可捞对象和地区；该选择不作为用户偏好保存，下次打开重置为默认。
- 树洞页：匿名倾诉、共鸣、树洞流、认证照片、安全提示。
- 广场页：公开动态、话题、热门内容、认证用户入口、附近入口。
- 附近的人页：粗略距离、真人认证、关注、好友申请。
- 游戏页：常规真心话、私密真心话、常规大冒险、私密大冒险。
- 钱包页：充值金币、收益金币、魅力值、提现规则、礼物、流水。
- 认证收益页：人脸/性别认证、拉新赠会员、收益规则、认证用户。
- 后台页：独立管理员入口，不出现在普通用户“我的”页面。
- 我的页：签到、人脸认证、拉新赠会员、黑名单、钱包、会员、记录。

认证用户不做单独公开列表。认证和性别只作为内容标签展示在广场帖子、附近的人、瓶子作者等业务卡片上。

## 筛选

以下功能统一支持城市、性别、年龄段筛选：

- 漂流瓶
- 广场
- 游戏
- 树洞
- 附近的人

筛选字段首版为前端状态，后续进入接口查询参数：`city`、`gender`、`age_range`。
## 2026-06-30 新规则：删除树洞、消息承接与匹配筛选

- 树洞功能从用户端删除，不再作为底部 tab、游戏入口、消息跳转目标或继续聊来源展示。
- 用户端底部导航调整为：`瓶子 / 广场 / 游戏 / 消息 / 我的`。其中“消息”承接原树洞位置，展示留言、系统通知、邀请卡片和真实私聊列表。
- 游戏页新增“随机匹配”入口，入口必须可点击跳转到匹配页或匹配面板；随机匹配必须支持性别筛选、年龄区间筛选和次数展示。
- 游戏随机匹配次数必须与项目其他次数体系保持一致：同一用户的剩余次数从统一 quota/用户状态读取，不能前端写死。
- 附近的人年龄筛选改为双端进度条，支持两端拖动选择年龄范围。
- 附近的人不再展示距离筛选入口。
- 所有涉及城市筛选的页面统一使用城市选择规则：默认显示 `全国 / 北京 / 上海 / 广州 / 深圳 / 全部`；点击“全部”展开全国其他省份/城市，并按页面空间调整展示个数与布局。
- 城市筛选必须可复用到广场、附近的人、随机匹配和后续所有需要城市过滤的列表。
- 头像规则：所有用户头像不准再显示文字头像。用户设置了头像时必须优先显示用户头像；未设置头像时使用系统随机头像池。头像池需来自合规免费图库或本地授权资源，并缓存到项目可控资源或后端假数据。
- 消息数据规则：用户端消息页展示的数据由后端数据库或测试数据库生成的假数据提供，前端不得只用本地硬编码假数据。
- 邀请消息必须使用卡片形式，至少包含来源、邀请人、房间/会话摘要、同意、取消。用户同意后创建或绑定房间；二次点击已同意的邀请卡片时直接跳转已有房间或会话详情。
- 数据同步规则：用户更新资料、头像、城市、性别、年龄后，附近的人、消息、匹配和聊天页必须读取最新用户资料；不得继续显示旧文字头像或过期派生数据。
