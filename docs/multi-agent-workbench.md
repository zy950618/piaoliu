# 多 Agent 协作入口

## 当前总目标

先搭建后台接口和独立 Web 管理后台，再继续细化前台体验和业务细节。后台不得挂在小程序、iOS、Android 或 H5 用户端路由中。

## 2026-06-23 继续完成阶段目标

本轮已从“文档登记”推进到“代码骨架落地”。当前已完成 Mock 鉴权、角色依赖占位、统一错误响应、审计链路、数据库/Alembic 占位和后台详情/批量操作入口；后续继续接入生产级持久化、真实会话和完整权限矩阵。

补充纠正：后台管理端已经从 `src/pages/admin` 剥离到独立 `admin-web/`。后续所有后台页面、后台布局、后台筛选和后台批量操作都在 `admin-web/` 中处理，用户端 `src/pages.json` 不再加入后台页面。

- 管理员真实鉴权骨架：登录、退出、当前管理员、未登录拦截和 token/session 处理边界。
- 角色权限：定义超级管理员、审核管理员、运营管理员、财务管理员的菜单和接口权限边界。
- 数据库/Alembic 占位：为 PostgreSQL、Redis、SQLAlchemy async、Alembic 迁移和种子管理员预留计划入口。
- 统一错误码：补齐未登录、无权限、参数错误、对象不存在、状态冲突、重复操作和系统错误的响应口径。
- 审计日志链路：把管理员操作、审核动作、配置变更、批量操作和错误响应纳入审计记录闭环。
- 后台详情/批量操作/高级筛选入口：为用户、内容、举报、提现、认证复核等模块补详情页、批量处理和高级筛选接力入口。

- [后续细节优化入口](detail-optimization-inbox.md)：收口后续体验、规则和实现细节，不阻塞后台骨架优先目标。

## 协作规则

- 多 Agent 可以并行处理代码或文档，但必须追加历史和状态，不要删除他人记录。
- 每一步处理后都要写入：要求、处理动作、当前状态、验证方式或待验证原因。
- 如果发现代码与文档不一致，先判断是否属于当前任务范围；范围内直接修正，范围外写入待核对项。
- 后台骨架优先级高于前台细节优化；前台细节统一进入后续优化入口。

## 交接格式

每个 Agent 完成任务后，在相关文档追加以下信息：

- 日期：使用 `YYYY-MM-DD`。
- 处理范围：明确页面、接口、后台模块或规则。
- 处理结果：说明新增、完善或仅确认。
- 验证：说明阅读、命令、截图或未验证原因。
- 后续入口：指向仍需处理的文档和条目编号。

## 当前接力建议

1. 后台接口 Agent：从 [接口与后台搭建计划](backend-interface-admin-plan.md) 的“继续完成阶段代码落地状态”开始，接入生产级账号、token/session、真实数据库和完整权限矩阵。
2. 后台管理 Agent：只在 `admin-web/` 中继续做管理后台，高级筛选、导出、趋势图和更多批量操作不得写回用户端 `src/pages/**`。
3. 产品规则 Agent：维护 [需求台账](requirements-ledger.md) 和 [处理历史](work-history.md)，确保每次新要求都有记录。
4. 数据持久化 Agent：按后台计划把 PostgreSQL/Redis/SQLAlchemy async/Alembic 从占位推进到真实连接、迁移执行和种子管理员。
5. 前端体验 Agent：只处理已明确进入 P1/P2 的体验问题，未排期细节放入 [后续细节优化入口](detail-optimization-inbox.md)。
