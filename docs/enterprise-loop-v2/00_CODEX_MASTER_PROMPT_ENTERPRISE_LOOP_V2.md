## 1. 最高强约束

### 1.1 项目业务不可变规则

1. 后台必须是独立 `admin-web/`，不得挂回用户端 `src/pages/**` 或 `src/pages.json`。
2. 用户端底部 Tab 固定：瓶子 / 广场 / 游戏 / 树洞 / 我的。
3. 禁止无上下文冷启动骚扰；允许明确互动上下文内的陌生人私聊，好友关系不是私聊唯一门槛。
4. 关注不等于好友。
5. 充值金币不可提现。
6. 收益金币转换为魅力值，提现按魅力值门槛和比例处理。
7. 私密照片采用 AI 智能审核优先、风险分级、人工复核兜底，收益只来自审核通过且未冻结内容。
8. 人脸认证包含活体检测、性别识别和人工复核；性别识别不能自动最终通过。
9. 附近的人只能展示粗略距离，不得展示精确位置。
10. 隐藏/仅主人可见留言不得向普通浏览者展示；给主人展示时必须匿名化作者身份。
11. 正常内容自动通过；举报、违规词、风控异常、私密照片进入审核。
12. 后台所有审核/配置/提现/封禁/批量动作必须写审计日志。

### 1.2 工程强约束

1. 不准全量重写。
2. 不准为了套模板破坏已有业务。
3. 不准用 Mock 冒充生产完成。
4. 不准只改 UI 不跑测试。
5. 不准删除历史文档和已有记录。
6. 不准跳过失败项。
7. 不准把“截图看起来正常”等同于“生产可用”。
8. 不准引入重依赖但不说明原因、包体影响、构建影响和跨端影响。
9. 不准修改敏感业务规则而不更新测试。
10. 不准把用户端做成后台表格，也不准把后台做成移动端玩具页面。

## 2. 本项目 LOOP 标准

每个任务必须使用：

```text
RPP-AOEVRR-D
Read → Protect → Plan → Act → Observe → Evaluate → Verify → Reflect → Record → Decide
```

每轮最多处理：

- 1 个 P0，或
- 2 个 P1，或
- 3 个 P2。

每轮必须先写计划，再动手。

### 2.1 开始模板

```text
【LOOP START】
目标：

禁止修改：
子 Agent：
预期验证：
人工门禁：
回滚方案：
```

### 2.2 结束模板

```text
【LOOP RESULT】
完成内容：

性能观察：

```

## 3. 多子 Agent 并行作业

必须拆分成以下 Agent。即使实际只有一个 Codex 实例，也要按这些角色分段工作，不能混在一起乱改。

| Agent | 名称 | 负责范围 | 主要输出 |
| --- | --- | --- | --- |
| A0 | Orchestrator | 读文档、拆任务、控制 LOOP、合并验收 | LOOP 计划、依赖图、最终验收报告 |
| A1 | Backend Identity/RBAC | 管理员、密码、token/session、权限矩阵 | admin/auth、roles、permissions、tests |
| A2 | Database/Persistence | PostgreSQL、Redis、Alembic、仓储层 | migrations、repositories、seed、持久化测试 |
| A3 | Admin Web UX | 独立 Web Admin、菜单、筛选、批量、详情、导出 | admin-web 页面、权限 UI、员工效率 |
| A4 | Moderation/Safety | 内容审核、举报、聊天审核、违规词、私密照片 | 审核队列、风控状态、审核证据 |
| A5 | Wallet/Finance | 金币、收益金币、魅力值、提现、财务风控 | 钱包/提现接口、财务后台、冻结规则 |
| A6 | User Frontend UX | 漂流瓶、广场、树洞、游戏、我的体验 | 用户端交互、状态、动效、转化路径 |
| A7 | Growth/Product | 扩列、房间、会员、礼物、拉新、活动 | 产品规则、接口草案、转化漏斗 |
| A8 | QA/E2E/Visual | 单测、契约、E2E、截图、回归 | 测试报告、截图、失败项 |
| A9 | Observability/SRE/Security | 日志、trace、metrics、SLO、安全基线 | telemetry、SLO、告警、安全检查 |
| A10 | Docs Keeper | 文档同步、需求台账、处理历史、验收清单 | docs 更新、交接记录 |

### 3.1 并行顺序

```text
Batch 0：A0 + A10 读取和冻结规则
Batch 1：A1 + A2 并行，A9 提前定义日志/trace/request_id
Batch 2：A3 + A4 + A5 并行，依赖 A1/A2 的权限和数据层
Batch 3：A6 + A7 并行，依赖产品规则和接口契约
Batch 4：A8 全量验证，A9 验证 SLO/安全/监控，A10 同步文档
Batch 5：A0 最终合并验收
```

### 3.2 冲突处理

- 如果两个 Agent 修改同一文件，必须由 A0 决定先后顺序。
- 如果 UI Agent 想改业务规则，必须转给 A7 和 A0。
- 如果 Backend Agent 改接口字段，必须通知 A3/A6/A8/A10。
- 如果 DB Agent 增加迁移，必须附带 downgrade 或兼容策略。
- 如果 QA 发现 P0 失败，所有新功能停止合并。

## 4. 第一阶段必须优先做：生产级后台底座

### 4.1 管理员真实鉴权

必须实现：

- `admins` 表。
- 密码哈希，不能明文存储。
- 禁用管理员不能登录。
- 登录失败次数限制。
- session/token 持久化。
- 退出后 token 失效。
- `GET /admin/auth/me` 返回角色、权限、可见菜单。
- 未登录统一 `AUTH_REQUIRED`。
- 无权限统一 `FORBIDDEN`。

验收：

- 未登录访问后台写接口失败。
- 禁用账号登录失败。
- 错误密码失败且不泄露用户是否存在。
- 登录后能访问允许菜单。
- 退出后旧 token 无效。
- 后台刷新页面后登录态可恢复或明确重新登录。

### 4.2 RBAC 权限矩阵

角色至少：

```text
super_admin
review_admin
ops_admin
finance_admin
support_admin
risk_admin
read_only_admin
```

权限维度：

```text
menu:view
list:read
detail:read
action:write
batch:write
export:write
config:write
finance:review
risk:freeze
audit:read
```

验收：

- 菜单权限和接口权限来源一致。
- 按钮不可见不等于安全，接口仍必须校验。
- 财务权限不能被审核员越权。
- 审核员不能修改奖励配置。
- 只读管理员不能执行任何写操作。

### 4.3 数据库与迁移

必须实现或补强：

- PostgreSQL 连接配置。
- Redis 连接配置。
- SQLAlchemy async session 生命周期。
- Alembic 迁移执行。
- seed 管理员。
- admin/roles/permissions/audit_logs/configs/moderation_records/withdrawals/verifications 等关键表。

验收：

- `alembic upgrade head` 可执行。
- 空库可启动。
- seed 管理员可登录。
- 重启服务后数据仍存在。
- 测试库与运行库隔离。
- 迁移不可破坏现有 SQLite 过渡数据说明。

### 4.4 审计日志真实落库

每条审计必须包含：

```text
audit_id
request_id
trace_id
operator_id
operator_role
module
action
object_type
object_id
before_state
after_state
reason
ip
user_agent
result
error_code
created_at
```

验收：

- 登录、退出、失败登录、权限拒绝、审核、封禁、解封、配置修改、提现审核、批量处理都有审计。
- 批量操作记录总结果和每个失败项。
- 审计详情页能关联用户、内容、提现、举报或认证记录。
- 审计不可由普通管理员删除。

## 5. 第二阶段：企业级 Web Admin

### 5.1 布局与信息架构

参考企业中后台，不套壳但要达到成熟结构：

```text
登录页
  └── 主框架
      ├── 顶部：管理员、角色、环境、通知、退出
      ├── 侧边栏：按角色显示菜单
      ├── 面包屑/页签
      ├── 内容区
      └── 全局消息/错误/加载
```

菜单：

```text
总览 Dashboard
审核工作台
  - 内容审核
  - 聊天审核
  - 私密照片审核
  - 认证复核
举报与风控
  - 举报队列
  - 拉黑记录
  - 风控命中
用户管理
钱包与提现
运营配置
  - 次数/广告/签到/拉新
  - 随机话术
  - 礼物/会员
数据分析
审计日志
系统设置
```

### 5.2 员工效率标准

后台不是展示页面，而是员工处理问题的工具。每个高频队列必须提供：

- 顶部 KPI：待处理、今日新增、超时、风险高、已处理。
- 高级筛选：状态、时间、城市、性别、年龄、风险、处理人、来源、金额。
- 批量操作：通过、拒绝、下架、冻结、转人工、关闭。
- 详情抽屉：用户、内容、历史、关联举报、关联聊天、审计记录。
- 快捷原因模板。
- 二次确认。
- 操作后自动跳到下一条。
- 键盘快捷键可选：J/K 上下条、A 通过、R 拒绝、Esc 关闭。

验收：

- 审核员处理一条内容不超过 3 次点击。
- 财务处理提现必须看到充值金币/收益金币/魅力值区分。
- 风控冻结必须要求原因。
- 每个处理动作后列表状态即时更新。

## 6. 第三阶段：用户端商业级体验

### 6.1 用户端不能像后台

用户端必须符合真实用户习惯：

- 第一屏只放核心任务。
- 文案短、明确、有情绪但不过度营销。
- 操作反馈即时。
- 错误提示要告诉用户下一步怎么办。
- 权限申请要解释价值，不要冷冰冰弹权限。
- 列表卡片要有真实信息层级，不能堆标签。
- 弹窗不能误关闭，重要输入不能误丢。
- H5、小程序、App 都要考虑触控目标。

### 6.2 页面级要求

#### 漂流瓶

- 保留捞/扔双入口。
- 捞瓶必须有明确动效和结果反馈。
- 扔瓶输入第一视觉，随机话术入口轻量。
- 筛选是辅助，不抢主视觉。
- 回应、举报、拉黑、关注都有明确反馈。
- 无匹配不扣次数。

#### 广场

- 信息流第一，发布入口明确但不遮挡。
- 图片/语音/视频真实展示。
- 点赞可切换，不能重复刷数。
- 留言弹窗轻，详情页完整。
- 隐藏留言隐私规则必须严格。
- 附近、礼物榜、新人推荐要可点击且有真实逻辑或明确 Mock 边界。

#### 树洞

- 强化匿名安全感。
- 发布前提示审核/匿名范围。
- 共鸣反馈温和，不像点赞攀比。
- 风险词命中时给温和提示，不直接羞辱用户。

#### 游戏

- 常规/私密模式区分明确。
- 抽题动效轻快。
- 私密模式必须有安全边界和内容尺度控制。

#### 我的

- 钱包、认证、签到、会员、拉新、黑名单层级清晰。
- 收益规则必须可理解。
- 认证流程展示进度和失败原因。

### 6.3 扩展功能：扩列、普通房间、私密房间

新增功能必须先写产品规则和接口契约，再写代码。

#### 扩列

目标：增强社交连接，不破坏“禁止无上下文冷启动骚扰，允许明确互动上下文内私聊”的规则。

功能：

- 扩列卡片。
- 兴趣标签。
- 城市/年龄/性别筛选。
- 关注/申请好友。
- 防骚扰限制。
- 举报/拉黑。

验收：

- 无明确互动上下文时不得发起冷启动私聊。
- 完成明确互动上下文与双向回应/确认后，可开启带 `source_type/source_id` 的临时私聊，无需先成为好友。
- 粗略距离，不展示精确位置。
- 拉黑后不能继续申请。

#### 普通房间

目标：多人轻互动。

功能：

- 创建普通房间。
- 房间主题。
- 人数上限。
- 公开/半公开。
- 加入/退出。
- 房主管理。
- 举报。

验收：

- 房间消息进入风控和审核链路。
- 房主踢人有审计或本地记录。
- 被拉黑用户不能进房。

#### 私密房间

目标：更强关系门槛，不做违规内容温床。

功能：

- 邀请制。
- 好友/认证/会员门槛可配置。
- 内容尺度规则。
- 截图/分享提醒。
- 举报和风控优先级更高。

验收：

- 私密不等于无监管。
- 未满足关系/认证门槛不能进入。
- 所有收益相关行为都必须可审计。

最终回答必须包含：

```text
1. 本轮做了什么
2. 多 Agent 分工结果
```
