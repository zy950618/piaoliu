# AI LOOP 标准与漂流瓶项目自定义 LOOP

版本：v2.0  
日期：2026-06-29

## 1. 先定义：AI 中的 LOOP 是什么

AI LOOP 不是简单地“让 AI 一直改到满意”。生产语境下，AI LOOP 是一个**受控智能体工作回路**：

```text
目标 Goal
  ↓
读取上下文 Read / Observe
  ↓
建立假设与计划 Plan
  ↓
执行动作 Act
  ↓
收集结果 Observe
  ↓
评价结果 Evaluate
  ↓
反思修正 Reflect
  ↓
更新状态 Record / Memory
  ↓
决定继续、回滚、暂停、交给人工或结束 Decide
```

它包含四类关键约束：

1. **状态约束**：每轮必须知道当前项目状态、已完成项、未完成项、风险项。
2. **动作约束**：每轮允许修改哪些文件、禁止修改哪些文件、是否需要迁移、是否需要人工审批。
3. **评价约束**：每轮必须跑哪些测试、截图、接口冒烟、性能、安全或人工体验检查。
4. **停止约束**：达到验收标准才停止；失败、证据不足、超出范围或破坏业务时必须暂停、回滚或转人工。

## 2. AI LOOP 的来源与可参考模式

本项目采用复合 LOOP，而不是单一名词：

| 模式 | 核心思想 | 本项目落地方式 |
| --- | --- | --- |
| ReAct | 推理和行动交替，通过外部工具/环境观察结果，再更新计划。 | Codex 每轮必须 Read → Plan → Act → Observe → Evaluate，而不是直接改。 |
| Reflection | 对上一步结果进行批判和修正，发现遗漏、错误、边界问题。 | 每轮必须写“失败项/风险项/下一轮修正”，不能只写成功。 |
| Human-in-the-loop | 对高风险工具调用或生产风险动作增加人工批准点。 | DB 迁移、权限矩阵、删除文件、支付/提现规则、隐私规则必须设置人工门禁。 |
| Evals / Regression | 用数据集、测试、CI 回归来衡量质量，而不是靠主观。 | 每个功能必须有单测/契约测试/端到端冒烟/视觉验收。 |
| SRE Feedback Loop | 以 SLI/SLO/错误预算反推工程优先级。 | 慢接口、错误率、审核积压、转化漏斗进入后台监控。 |
| Product UX Loop | 以用户任务完成率、摩擦点、留存和反馈改进体验。 | 每轮用户端优化必须检查真实人的理解、点击路径、等待反馈和失败恢复。 |

## 3. 为什么当前项目不能用“普通 LOOP”

漂流瓶项目涉及：

- 匿名社交和隐私。
- 私密照片、收益、魅力值、提现。
- 审核、举报、聊天记录、违规词、风控。
- 用户端 H5/小程序/App 与独立 Web Admin 双端。
- 后台员工处理效率和审计责任。

因此 LOOP 必须带有**生产级风控门禁**，不能只做页面美化。

## 4. 漂流瓶项目的企业级 LOOP 定义

本项目采用 `RPP-AOEVRR-D` 十二步 LOOP：

```text
R  Read          读取项目文档、代码、历史、当前任务
P  Protect       提取不可变业务规则和禁止修改区
P  Plan          拆计划、拆 Agent、拆验收标准
A  Act           按最小变更单元实现
O  Observe       运行、截图、接口冒烟、日志观察
E  Evaluate      对照门禁打分，判断是否通过
V  Verify        自动化测试 + 人工路径验证 + 视觉验证
R  Reflect       记录失败项、返工项、风险项、改进项
R  Record        更新 work-history / checklist / ledger / API / design docs
D  Decide        继续下一轮、回滚、暂停等待人工、或交给其他 Agent
```

## 5. 每轮 LOOP 强制输入

每轮开始前，Codex 必须写出：

```text
本轮目标：
本轮优先级：P0 / P1 / P2
已读取文件：
当前业务不可变规则：
允许修改范围：
禁止修改范围：
涉及 Agent：
需要人工审批的动作：
预期验收证据：
失败回滚方案：
```

没有这段，不允许进入实现。

## 6. 每轮 LOOP 强制输出

每轮结束必须写出：

```text
本轮实际修改文件：
新增/修改接口：
新增/修改数据库迁移：
影响的用户路径：
影响的管理员路径：
执行过的命令：
测试结果：
接口冒烟结果：
截图/录屏证据：
失败项：
未完成项：
风险：
文档同步位置：
下一轮建议：
是否允许合并：YES / NO
```

## 7. LOOP 停止条件

只有满足以下条件，才允许停止并声明完成：

1. 本轮目标全部通过验收。
2. 自动化测试通过。
3. 关键路径手工验证通过。
4. 没有违反不可变业务规则。
5. 文档同步完成。
6. 没有 P0/P1 阻塞项。
7. 证据链完整。

## 8. LOOP 失败条件

出现任意一项，必须停止并进入修复或回滚：

- 未读项目文档直接改代码。
- 后台被重新挂到用户端 `src/pages/**`。
- 私聊、提现、私密照片、隐藏留言、认证复核等规则被破坏。
- 使用 Mock 冒充生产级实现。
- 只改 UI 不跑测试。
- 测试失败仍声明完成。
- 未同步 `work-history.md`、`completed-checklist.md`、`requirements-ledger.md`。
- 没有写失败项和下一轮风险。

## 9. LOOP 分级

| LOOP 类型 | 用途 | 执行频率 | 负责人 |
| --- | --- | --- | --- |
| Micro LOOP | 单文件/单函数/单组件改动 | 每个 Agent 内部多次 | 子 Agent |
| Feature LOOP | 一个功能闭环，如后台 RBAC、审核详情、广场留言 | 每个任务一次 | 功能 Agent |
| Release LOOP | 一组功能合并和发布验收 | 每个阶段一次 | Orchestrator + QA |
| Product LOOP | 用户反馈、数据指标、运营目标再规划 | 每周或每版本 | Product/UX Agent |
| Safety LOOP | 安全、隐私、风控、审计复核 | 每次涉及敏感能力 | Safety Agent |

## 10. 本项目推荐 LOOP 批次

### Batch 0：读取与冻结规则

- 读取所有 docs。
- 读取 `admin-web/`、`src/`、`backend/`、`backend/alembic/`。
- 生成不可变规则清单。
- 生成当前 Mock/真实能力边界清单。

### Batch 1：生产级后台底座

- 真实管理员账号。
- 密码哈希。
- token/session。
- RBAC。
- PostgreSQL/Redis。
- Alembic。
- 审计真实落库。

### Batch 2：员工效率后台

- 高级筛选。
- 批量处理。
- 审核工作台。
- 导出权限。
- 趋势图。
- 队列 SLA。

### Batch 3：用户端体验与商业化

- 漂流瓶、广场、树洞、游戏、我的体验统一。
- 加载/空态/错误/审核/慢接口统一。
- 扩列、普通房间、私密房间。
- 私密照片闪图、礼物、会员、拉新闭环。

### Batch 4：监控与增长闭环

- OpenTelemetry。
- SLO。
- 用户留存和转化漏斗。
- 审核积压监控。
- 风控预警。

## 11. AI LOOP 的验收不是一句“通过”

每个通过项必须有证据：

| 验收类型 | 证据 |
| --- | --- |
| 代码正确 | 测试命令和结果 |
| 接口正确 | 请求、响应、状态码、错误码 |
| UI 正确 | 截图或 Playwright 断言 |
| UX 正确 | 用户任务路径和失败恢复 |
| 后台效率正确 | 队列处理耗时、批量结果、审计记录 |
| 权限正确 | 不同角色可见/不可见、可操作/不可操作证据 |
| 数据正确 | 迁移、种子、读写、重启后仍存在 |
| 安全正确 | 未授权、越权、重复提交、敏感字段隐藏证据 |
| 文档正确 | 修改了哪些文档，新增了哪些历史记录 |
