# 漂流瓶多回应者与双人回复组设计

日期：2026-07-17

## 目标

保留当前瓶主可查看多位回应者的漂流瓶回复页，在每位回应者下方补齐瓶主与该回应者之间可持续往来的双人回复组、回复输入和未读状态。

本轮只处理漂流瓶回复链。头像资料弹窗、点赞、礼物、好友申请和普通私聊入口在下一轮单独设计，不与回复链同时实现。

## 一、隐私边界

漂流瓶回复不是公开评论。

- 瓶主可以查看该瓶子收到的全部回应者及各自双人回复组。
- 单个回应者只能查看自己与瓶主之间的回复组。
- 回应者不能通过前端参数、分页接口或回复ID查看其他回应者的内容。
- 服务端必须同时校验 `bottle_id`、`owner_id`、`responder_id` 和当前登录用户，前端隐藏不能代替权限校验。
- 普通个人私聊与漂流瓶回复组是不同实体，不能共用消息记录或会话ID。

## 二、瓶主聚合视图

### 页面结构

页面继续显示：

1. 原漂流瓶内容。
2. “收到的回应”数量。
3. 多位回应者的回复组，按最近活动时间排序。

每个回应者占一个连续区段，不使用独立悬浮卡片。区段之间使用分隔线和留白区分。

### 回应者区段

每个区段包含：

- 回应者头像、昵称、原始回应时间。
- 原始回应内容。
- 瓶主与该回应者的后续往来。
- 当前回应者的“回复”入口。

页面默认展开所有回应者区段，不把多位回应者合并成公共对话，也不要求瓶主先进入单独线程页才能阅读。

## 三、双人回复层级

### 展示顺序

单个回应者区段按时间顺序展示：

```text
小雨的原始回应
  我 回复 小雨
  小雨 回复 我
  我 回复 小雨
```

### 视觉层级

- 原始回应保持现有头像加正文布局。
- 后续往来统一缩进一层，缩进不随回复深度继续增加。
- 每条后续消息显示“我 回复 小雨”或“小雨 回复 我”。
- 瓶主消息使用极浅海蓝背景；回应者消息使用产品白或浅灰背景。
- 不使用无限树状缩进、连线树、气泡套气泡或多层卡片。
- 同一回复组中的全部消息始终只属于瓶主与该回应者。

### 回复目标

点击某条消息的“回复”或区段底部“回复”后：

- 页面底部出现回复输入栏。
- 输入栏显示“回复小雨”或当前目标用户。
- 发送的新消息写入该回应者对应的 `thread_id`。
- 发送成功后追加到当前区段底部并保持可见。
- 取消回复后关闭输入栏，不清除已经输入的草稿，除非用户主动清空或离开页面。

## 四、未读状态

### 展示

- 有回应者新消息时，在该回应者昵称旁显示一个小型操作蓝圆点。
- 圆点只表达当前瓶主尚未查看的回复组消息，不显示未读数字。
- 瓶主自己的回复不会产生该区段的未读点。

### 清除条件

满足以下任一条件时，当前回应者区段标记为已读：

1. 从该回应者的互动通知直接进入，目标消息完成定位并展示。
2. 瓶主点击该回应者区段或区段中的消息。
3. 瓶主点击“回复”进入该回应者的回复输入状态。
4. 瓶主成功向该回应者发送回复。

仅打开普通瓶子详情但未查看或操作该回应者区段时，不清除未读点。

已读状态必须由接口持久化；刷新页面后不能恢复为未读。

## 五、数据模型

### BottleResponseThread

每个瓶主与回应者针对同一瓶子只有一条回复组：

- `thread_id`
- `bottle_id`
- `owner_id`
- `responder_id`
- `created_at`
- `last_message_at`
- `owner_unread_count`
- `owner_last_read_message_id`
- `responder_unread_count`
- `responder_last_read_message_id`
- `status`

数据库需要对 `(bottle_id, owner_id, responder_id)` 建立唯一约束，避免弱网重复创建回复组。

### BottleReplyMessage

- `message_id`
- `thread_id`
- `sender_id`
- `reply_to_message_id`
- `content_type`
- `content`
- `created_at`
- `moderation_status`
- `deleted_at`

`reply_to_message_id` 用于显示“回复某人”和审计关系，不用于生成无限视觉缩进。

## 六、权限与关系

- 回应者首次回应瓶子后，瓶主获得来源为 `content_reply` 的定向私聊授权。
- 回应者不会仅因自己回应瓶子就自动获得对瓶主的普通私聊权限。
- 瓶主在回复组内回复该回应者后，该回应者获得对应的 `content_reply` 授权。
- 拉黑关系会禁止继续发送，但不应让历史漂流瓶回复错误暴露给第三方。
- 删除普通个人会话、清空聊天记录或退出房间都不能删除漂流瓶回复组。

## 七、异常与边界

- 回应者账号受限：历史内容按审核结果展示，回复输入禁用并显示系统说明。
- 某条消息被删除：保留时间顺序位置，显示“这条回复已不可见”。
- 瓶子被删除或违规下架：关闭继续回复能力，保留合规要求允许展示的历史记录。
- 发送失败：保留输入内容，显示重试入口，不追加成功消息。
- 重复发送：使用幂等键，不能创建两条相同回复。
- 回应者数量较多：页面按回复组分页加载，单个回复组内按消息时间分页。

## 八、验证标准

### 瓶主视角

- 同一页面能看到多位回应者。
- 每位回应者下方只出现与瓶主的双人往来。
- 瓶主之前发给小雨的回复能够按时间和层级展示。
- 点击小雨“回复”后输入栏明确显示当前目标。
- 发送后消息只追加到小雨区段。

### 回应者视角

- 小雨只能看到自己与瓶主的回复组。
- 修改URL参数或回复ID无法读取其他回应者的数据。
- 瓶主回复小雨后，小雨可以继续回复瓶主。

### 未读

- 小雨有新消息时昵称旁显示蓝色未读点。
- 点击小雨区段、进入回复状态或成功回复后未读点消失。
- 操作其他回应者不会清除小雨的未读状态。
- 刷新后已读状态保持。

### 通用

- 390×844 手机视口下层级清楚且无横向溢出。
- 回复输入不遮挡正在编辑的回应组。
- 减少动态效果模式下定位和输入栏出现不使用强制滚动动画。
- 浏览器控制台无 JavaScript 错误和警告。
