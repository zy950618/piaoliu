# 漂流瓶项目｜新会话细节处理交接摘要

更新时间：2026-07-20

## 新会话目标

在不改变已经确认的产品逻辑与交互结构的前提下，继续对微信小程序原型做细节级视觉、交互和状态验证。每次只修改用户明确指出的区域，修改后必须回归验证相邻功能，禁止顺手重构或改变已经确认的设计。

## 产品主线

- 产品不是单一漂流瓶，而是“遇见 / 广场 / 玩法产生关系 → 消息承接关系”的轻社交产品。
- 遇见以匿名漂流瓶和真诚回应为核心；附近、广场、玩法辅助建立关系。
- 消息中心负责聚合通知、申请、个人会话和房间会话。
- 所有设计必须符合微信小程序和真实手机操作习惯，视觉采用已确认的“潮汐白页”浅色体系：清透、圆润、克制、有层次，避免黑底、刺眼色块、方格化和过度装饰。

## 已确认的消息与关系逻辑

### 消息中心

- 顶部三个聚合入口：互动通知、邀请与申请、房间列表；只通过角标提示未读。
- 下方按用户或房间展示最近会话。
- 消息免打扰在会话列表显示静音图标。
- 邀请与申请支持左滑删除。
- 漂流瓶回复通知应跳转到对应瓶子的具体回复位置，不跳到普通聊天。

### 漂流瓶私密回复

- 一个瓶子可收到多人回复，但每位回复者与瓶主人形成独立、仅双方可见的回复线。
- 瓶主人页面可展开全部回复者分组；只有对应回复者的新内容显示蓝色未读点。
- 瓶主人点击或回复后，该回复线未读点消失。
- A、B 可以在瓶子回复区持续往返回复，不自动转成普通私聊。
- 回复按钮只显示“回复”，不出现“回复某某”。
- 点击用户头像打开资料弹窗，包含点赞、送礼物、私聊、好友申请等入口。
- 私聊权限：必须由对方先回复自己的内容；按钮灰色时仍可点，并提示“对方回复后才可以私聊”。拉黑后保留权限记录，但发送入口被阻断并提示已被拉黑。

### 好友与会话状态

- “聊过”“私密聊”“已经是好友”是三个独立状态。
- 好友申请需一方申请、另一方同意，不区分男女。
- 申请默认 3 天过期；每日最多申请 10 次；滚动 10 天内对同一用户最多申请 3 次。
- 用户可关闭“允许别人申请好友”，默认开启。
- 不能因多人拒绝直接封号；拒绝、举报等只能进入风控评分与人工/规则审核。
- 清空记录、删除会话、拉黑用户、退出房间是完全不同的动作。
- 个人会话可清空、删除、拉黑；房间会话额外提供退出房间。
- 退出房间无需审批，退出者本地记录删除且不能再进入；其历史消息仍保留给其他成员。

### 房间

- 房间可公开或私密，可一对一或多人。
- 公开房允许任何人加入，或由房主开启“加入需确认”；房主有踢人权限。
- 私密房通过邀请加入；拒绝邀请后不产生房间会话，并通知邀请者“XX 拒绝进入 XX 房间”。
- 只有聊过的用户才能被邀请；广场中同样只能邀请聊过的好友/会话对象。
- 创建成功后立即进入最近会话，并出现在房间列表。
- 私密房内容尺度可比公开房宽松，但仍必须遵守平台安全与法律底线；公开房执行严格内容规范。
- 房间是聊天会话，只支持图片、闪照、视频等媒体；真心话、大冒险、骰子等通过聊天工具发起玩法消息。
- 默认长期存在；房主可解散；连续一个月无聊天记录自动解散。公开房与私密房在消息列表显示标签。

## 已确认的聊天输入区

- 单人聊天与房间聊天保持一致的输入结构。
- 语音按钮位于输入框内部，点击后切换“按住说话”；录音时展示声纹。
- 输入框右侧是发送按钮，再右侧是加号。
- 点击加号后输入区整体上移，下方展示工具宫格；不显示“媒体 / 互动”分组标题。
- 单人聊天工具保持：图片、闪照、视频、摇骰子、真心话、大冒险、房间、更多玩法。
- 房间工具保持已确认范围，不得因处理其他细节擅自增删。
- 调用真实手机输入法，不在正式实现中模拟自制键盘；原型仅用于表达键盘顶起和收起后的页面状态。

## 已确认的回复输入体验

- 点击回复后，当前页面内容和底部输入框随系统键盘自然上移。
- 回复框左侧有 X，可取消当前回复。
- 发送成功后：系统键盘收起、回复输入框收起、页面回到当前内容底部，不额外制造超长空白，也不跳到新页面。

## 玩法交互

- 真心话、大冒险点击后使用居中弹窗；可随机题目或自己输入。
- 随机按钮位于题目输入框内部右侧。
- 摇骰子必须展示骰子旋转/滚动后停止的过程，结果用骰面呈现，不使用纯文字替代。
- 当前真心话/大冒险弹窗已按用户确认的 A 方案调整为经典 iOS/微信式轻弹窗：白色圆润卡片、胶囊分段、清晰输入区、克制按钮。

## 必须遵守的修改边界

1. 工作区已有大量未提交修改与删除状态，全部视为用户资产；禁止 reset、checkout、还原或批量整理。
2. 每次先截图/复现，再只改目标区域；不能因为调整一个弹窗而改变聊天加号、语音、房间、漂流瓶回复等已确认逻辑。
3. 文案要像真实产品界面，不写开发解释、后台说明或“符合偏好”等用户不关心的提示。
4. 视觉细节以圆润、留白、清透、浅色层次为主；避免黑色大底、强渐变、粗重阴影、刺眼危险色和机械方格。
5. 每次修改后至少验证：目标交互、相邻入口、未读状态、输入框收起/展开、加号面板、语音入口。
6. 先完成逻辑完整性，再做审美 LOOP；不可用视觉改动掩盖状态或业务缺失。
